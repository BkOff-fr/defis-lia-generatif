# Chantier C29 — Polish Mode Équipe (v0.7.1)

> **Version cible** : v0.7.1 (patch release)
> **Sprint** : court, 2-3 jours
> **Pré-requis** : v0.7.0 shippée (C28 Phase 2 Implemented), 8 IPC `team_*` déjà câblés côté Rust mais sans UI côté frontend Tauri.
> **Objectif** : compléter UX Mode Équipe + outillage admin avant d'attaquer la candidature data.gouv.fr en v1.0.

---

## 0. Pourquoi ce patch

v0.7.0 a livré le binaire `sobria-team-aggregator` + extension navigateur Mode Équipe + 8 IPC Rust côté app. Mais 4 manques honnêtement notés dans la section "Différé v0.7.1+" du CHANGELOG :

1. **Frontend Tauri** : la section `/parametres → Mode Équipe` côté `web/` n'a pas été câblée. Les 8 IPC `team_*` sont fonctionnels en Rust mais l'utilisateur de l'app Tauri ne peut pas les activer depuis l'UI — il doit éditer la SQLite à la main (inacceptable).
2. **`admin reset-password`** : si l'admin oublie son mot de passe, il doit recréer la base ou éditer SQLite directement. Frustrant.
3. **`serve --regen-cert` / `--regen-key`** : impossible de régénérer le cert TLS sans recréer la base.
4. **Alertes seuils** : table `alert_thresholds` mentionnée dans le brief C28 mais reportée. Utile pour cas d'usage CSRD ("alerte si > 100 gCO₂eq/utilisateur/mois").

v0.7.1 ferme ces 4 manques. C'est un patch release, pas un saut majeur.

### C29.2 — CLI `admin reset-password` (0.5 jour)

**Livrable** : commande `sobria-team-aggregator admin reset-password <username>` qui prompt un nouveau password + hash Argon2id + update SQLite.

- `crates/sobria-team-aggregator/src/commands/admin.rs` :
  ```rust
  pub fn reset_password(state: &ServerState, username: &str) -> Result<()>;
  pub fn list_admins(state: &ServerState) -> Result<Vec<AdminSummary>>;
  ```
- CLI clap `admin {reset-password|list}` sous-commande de `sobria-team-aggregator`.
- `reset-password` :
  - Lit username, prompte password 2 fois (`rpassword` crate) avec confirmation.
  - Hash Argon2id PHC + UPDATE `admins SET password_hash = ?, last_login_at = NULL WHERE username = ?`.
  - Révoque tous les tokens admin actifs (`UPDATE tokens SET revoked_at = now() WHERE admin_id = ?`).
  - Sortie : "Mot de passe de <username> réinitialisé. 3 token(s) révoqué(s)."
- `list` : affiche un tableau `id | username | created_at | last_login_at`.
- Tests `tests/cli_admin.rs` : crée admin, reset-password, vérifie hash changé + tokens révoqués.

### C29.3 — `serve --regen-cert` + rotation TLS (0.5 jour)

**Livrable** : option CLI pour régénérer cert auto-signé sans recréer la base.

- `crates/sobria-team-aggregator/src/commands/serve.rs` :
  - Nouveau flag `--regen-cert` : avant de lancer le serveur, regen `cert.pem` + `key.pem` via `rcgen` (même CN, même SANs, validité 10 ans), backup old en `cert.pem.bak.<timestamp>`.
  - Affiche le fingerprint SHA-256 du nouveau cert (utile pour que les clients acceptent le nouveau).
- Section `docs/operations/team-aggregator.md` enrichie : "Rotation TLS" avec procédure (regen → notifier les utilisateurs du nouveau fingerprint → ils ré-acceptent dans extension/app).
- Tests `tests/regen_cert.rs` : démarre serveur, regen, vérifie cert différent + même CN.

### C29.4 — Alertes seuils mail/webhook (1 jour)

**Livrable** : table `alert_thresholds` + check au POST estimations + notifications optionnelles.

- Migration SQLite v2 :
  ```sql
  CREATE TABLE alert_thresholds (
      id TEXT PRIMARY KEY,                  -- ULID
      scope TEXT NOT NULL CHECK (scope IN ('user', 'team')),
      target_id TEXT,                       -- user_id si scope=user, NULL si scope=team
      period TEXT NOT NULL CHECK (period IN ('daily', 'weekly', 'monthly')),
      gco2eq_max REAL NOT NULL,
      notify_kind TEXT NOT NULL CHECK (notify_kind IN ('webhook', 'email', 'log_only')),
      notify_target TEXT,                   -- URL si webhook, email si email
      created_by_admin_id TEXT NOT NULL REFERENCES admins(id),
      created_at TEXT NOT NULL,
      disabled_at TEXT
  );

  CREATE TABLE alert_triggers (
      id TEXT PRIMARY KEY,
      threshold_id TEXT NOT NULL REFERENCES alert_thresholds(id),
      period_start TEXT NOT NULL,
      period_end TEXT NOT NULL,
      observed_gco2eq REAL NOT NULL,
      triggered_at TEXT NOT NULL,
      notified_at TEXT,
      notify_error TEXT
  );
  ```
- API admin :
  - `POST /api/v1/admin/alerts {scope, target_id?, period, gco2eq_max, notify_kind, notify_target?}`
  - `GET /api/v1/admin/alerts` → liste.
  - `DELETE /api/v1/admin/alerts/:id`.
  - `GET /api/v1/admin/alerts/triggers?from&to` → historique déclenchements.
- Logique de vérification :
  - À chaque `POST /api/v1/estimations`, calculer le total `gco2eq_p50` du user pour la période courante (daily/weekly/monthly).
  - Si un threshold actif pour ce user (ou team) est dépassé ET aucun trigger pour cette période → insérer `alert_triggers` + envoyer notification.
- Notifications :
  - `webhook` : POST JSON `{threshold, period, observed_gco2eq, scope, target_id}` à l'URL. Timeout 5 s.
  - `email` : SMTP via `lettre = "0.11"` (config `smtp_url` + `smtp_from` dans table `config`). Optionnel — si non configuré, fallback log only.
  - `log_only` : juste `tracing::warn!`.
- UI dashboard admin `web-team/src/routes/admin/alerts/+page.svelte` :
  - Table thresholds + bouton "Créer une alerte" (form avec scope, target user, period, gCO₂eq max, notify_kind, target).
  - Historique des 50 derniers triggers.

## 4. Risques + mitigations

| Risque | Probabilité | Mitigation |
|--------|-------------|------------|
| L'IPC `team_push_estimation` n'est pas appelé automatiquement depuis l'app Tauri | Moyenne | Vérifier l'intégration dans le dispatcher `sobria-app::logic::record_estimation` — peut-être déjà fait en C28.6, sinon C29.1 le câble. |
| Notification email échoue (SMTP mal configuré) | Haute | Fallback `log_only` automatique + bouton "Tester la notification" dans le dashboard alerts. |
| Regen cert casse les sessions clients actives | Haute | Doc claire + warning UI "Regenerate cert force les utilisateurs à re-accepter le fingerprint". |
| Alertes en cascade (1 dépassement → 1000 notifications) | Faible | Single trigger par period (UNIQUE sur threshold_id + period_start). |
